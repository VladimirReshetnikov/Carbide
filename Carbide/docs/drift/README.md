## Documented differences (M1)

Carbide inherits Mono-WASM's single-threaded model and the post-publish shape of the runtime. Two M1-specific deviations from "a dotnet CLI running locally" are worth surfacing to users:

- **Newline is `\n`, not `\r\n`.** `Console.WriteLine` inside Mono-WASM emits `"\n"` regardless of the host OS, because `Environment.NewLine` is `"\n"` on WASM. Carbide does not rewrite output. Tests and fixtures should expect `"\n"`.
- **AppDomain-less assembly caching.** Each call to `Project.run()` does `Assembly.Load(byte[])` and invokes the entry point. Mono-WASM has no AppDomain isolation, so types loaded by one run linger for the life of the process. Two consecutive runs with the same assembly name may see stale state. Users who need isolation between runs should dispose and reinitialise the session.
- **Trimming disabled in M1.** `IsTrimmable=false` and `PublishTrimmed=false` are set on `Carbide.Core.csproj` because the trimmer strips BCL methods that the user's compilation expects (e.g. `Console.Write`). M3 restores trimming once `@carbide/refs-net10.0` ships untrimmed reference assemblies that feed Roslyn, keeping the runtime trimmed. Tracked in [architecture §13 Q.3](../planning/carbide-architecture-and-implementation-plan__2026-04-17__16-16-47-000000.md#13-open-questions-architecture-positions).
- **Node metadata fetches go over HTTP, not file://.** The `NodeHostAdapter` starts a localhost asset server by default because Mono-WASM's HttpClient cannot stream a file:// response body reliably (reports 0 content-length and truncates reads). The browser host serves from same-origin HTTP naturally. Revisit when the Mono-WASM file:// fetch shim exposes bytes as a proper stream.

## Documented differences (M2)

- **Document paths are byte-for-byte opaque identity.** Carbide does not lower-case, canonicalise slashes, or trim whitespace from the `path` argument to `addSource` / `updateSource` / `removeSource`. `"helper.cs"` and `"Helper.cs"` are two different documents; diagnostics echo whatever string the caller passed. Windows-vs-Unix casing is not normalised because there is no correct answer. [M2 D17.]
- **One top-level-statements file per project.** Carbide inherits Roslyn's CS8802. If two user-added files contain top-level statements, compilation fails and the diagnostic names the offending file. The hidden `Carbide.GlobalUsings.g.cs` document contains only `global using` directives, which do not count as top-level statements. [M2 D19.]
- **`Carbide.GlobalUsings.g.cs` is a reserved path.** Attempting to `addSource` / `updateSource` / `removeSource` that exact path throws. The document is owned by Carbide's implicit-usings machinery and not exposed to callers. [M2 D18.]
- **`addSource` on a duplicate path throws; `removeSource` on an unknown path is a no-op.** Accidental double-add is almost always a bug and silent overwrite hides it. Removal tolerates missing paths so teardown code doesn't need a try/catch per file. `updateSource` throws on unknown paths. [M2 D15, D16.]

## Documented differences (M3)

- **Reference bytes are validated synchronously.** `session.addReference(bytes, name)` throws before returning a handle if the bytes lack a CLR metadata directory. Roslyn's later CS0009 "PE image doesn't contain managed metadata" diagnostic is the slow path; Carbide prefers the fast one. [M3 D29.]
- **References are session-scoped; handles encode `sessionId`.** Attaching a handle from session A to a project in session B throws synchronously at the TS layer; the C# side double-checks. Handles become `disposed: true` after `session.removeReference` or `session.shutdown`. [M3 D26, D28.]
- **Base64 at the JSExport boundary, bytes in the public API.** The TS surface accepts `Uint8Array` (architecture Q.2). The C# JSExport receives base64-encoded strings and decodes internally — cheaper marshalling than `Uint8Array` today. [M3 D23, D24.]
- **`@carbide/refs-net10.0` is an opt-in sibling package.** When installed (postinstall generates `ref/net10.0/*.dll` from the pinned `Microsoft.NETCore.App.Ref` nupkg), Carbide uses it as the compile-time API surface instead of the runtime's trimmed BCL. When absent, Carbide falls back to the runtime DLLs (M1/M2 behaviour). No DLLs are committed to the repo. [M3 D31, D32, D34.]
- **Ref-pack wins: no merging with the runtime.** When the ref-pack is available, Carbide feeds Roslyn *only* the ref-pack DLLs. Mixing ref-pack (untrimmed reference assemblies) and runtime (trimmed implementations) produces duplicate type definitions and CS0518 "Predefined type not defined" errors. The runtime continues to hold its own DLLs for execution. [M3 `boot.ts` `resolveRefPackUrls` / InitAsync call site.]
- **Attached references are pre-loaded before the user's PE runs.** Mono-WASM's default AssemblyLoadContext doesn't reliably resolve Assembly.Load(byte[])'d dependencies lazily, so Carbide calls `Assembly.Load` on every attached reference and registers an `AppDomain.AssemblyResolve` handler before loading the user's PE. The handler is removed in `finally` after the entry point returns. [M3 `ProjectCompiler.RunAsync`.]
- **Trimming stays off in M3 despite the ref-pack.** The architectural position (Q.3) calls for trimmed runtime + untrimmed ref-pack, but the trimmer decides what to keep based on Carbide.Core's own usage, not the user's. Re-enabling `<PublishTrimmed>true</PublishTrimmed>` would still risk stripping BCL members the user's emitted PE binds at JIT time. The ref-pack now decouples the *compile-time* surface from runtime trim state — that's the M3 win — but restoring the runtime trim requires ILLink work deferred to a later milestone. [M3 plan §5 D35.]

## Documented differences (M4)

- **`build()` auto-selects OutputKind.** When any syntax tree has top-level statements the compilation uses `ConsoleApplication`; otherwise `DynamicallyLinkedLibrary`. Libraries without a `Main` build; top-level apps build; `run()` always requires `ConsoleApplication`. [M4 plan updated in M5 after CS8805 surfaced for csproj-driven top-level apps; see `ProjectCompiler.InferOutputKind`.]
- **Source text is UTF-8 with encoding.** `SourceText.From(code, Encoding.UTF8)` is required so portable-PDB emission can record offsets. Without encoding Roslyn raises CS8055 "Cannot emit debug information for a source text without encoding." [M4 `ProjectCompiler.AddSource`/`UpdateSource`.]
- **PE and PDB bytes cross the JSExport boundary as base64.** Same rationale as the reference-upload path (M3 D24): JSExport's string pipeline is reliable; raw `Uint8Array` marshalling has edge cases. 33% overhead is acceptable for typical assembly sizes. [M4 D38.]
- **The CLI redirects `console.info` / `console.debug` to stderr.** Carbide's C# logger writes info-level messages to `globalThis.console.info`, which Node routes to stdout. The CLI's bin entry monkey-patches `console.info` and `console.debug` to write to stderr before the runtime boots, so `carbide build --format json` can emit pure JSON on stdout. The user program's own `Console.WriteLine` is captured separately and routed through the CLI's output formatter. [M4 `packages/cli/src/bin/carbide.ts`.]
- **CLI boots a fresh session per invocation.** No daemon or process pool. Each `carbide` call pays the cold-start cost (~3-5 s warm, more on first use). Scripts that loop over many invocations should switch to the `@carbide/core` API. [M4 D46; plan §6 R33.]
- **CLI exit codes are 0 (success) / 1 (compile errors) / 2 (i/o-or-internal) / 3 (bad flags) / N (for `run`, the user program's exit code).** Scripts that care only about pass/fail can continue to treat anything non-zero as failure. [M4 plan §5; `packages/cli/README.md`.]
- **`--out -` on `build` writes PE bytes to stdout (no PDB).** Useful for piping into downstream tools. Both `run` and `validate` leave stdout for their structured output formats. [M4 D47.]
- **Program arguments after `--` go to Main(string[]).** The parser treats `--` as a hard separator; `carbide run --source P.cs -- foo bar` passes `["foo", "bar"]` to the program. (As of M4, `ProjectCompiler.RunAsync` does not yet propagate program argv — this is a follow-up item called out in the M4 plan §8.)

## Documented differences (M5)

- **Deterministic builds are on by default.** `CSharpCompilationOptions.WithDeterministic(true)` makes two builds with the same inputs produce byte-identical PE. No runtime knob to turn this off — the acceptance that `carbide build --project` matches `carbide build --source` flattened-flags needs this. [M5 D53.]
- **JSON schema version bumped to 2.** `ProjectOptions` gained `defineConstants`; wire consumers that still send `schemaVersion: 1` work because the C# validator accepts 1 or 2. TypeScript clients from M5 onwards always send 2. [M5 D62.]
- **`.csproj` subset is bounded.** Only the properties listed in `@carbide/msbuild-lite`'s README are read. Unsupported constructs surface as `MSBLITE001` / `MSBLITE011` warnings; `PackageReference` and `ProjectReference` are captured but not resolved (`MSBLITE013` / `MSBLITE014`). `Directory.Build.props`, `<Import>`, property functions, `<Target>` / `<Task>` are not read at all. [M5 D55, D56, D57; §7.]
- **Compile-item paths are relative to the project directory, forward-slash-normalised.** Diagnostics show `Helpers/Thing.cs` (short, cross-platform), not absolute paths. The underlying `ProjectModel` keeps absolute paths in `sourceFiles` for the CLI's convenience. [M5 D51.]
- **First-listed `TargetFramework` wins.** `<TargetFrameworks>net8.0;net10.0</TargetFrameworks>` → `net8.0`. Multi-TFM output isn't a Carbide feature; callers who want a different TFM edit the csproj or override via `--configuration`. [M5 D50.]
- **Default Compile include / exclude set matches `cs_kit`.** `**/*.cs` rooted at the project directory; fixed excludes `bin/**`, `obj/**`, dotted directories (`.*`). `<EnableDefaultCompileItems>false</EnableDefaultCompileItems>` switches off the default include; explicit `<Compile Include="…">` is then required. [M5 D52.]
- **`--project` and `--source` are mutually exclusive at the CLI.** Passing both exits with code 3. `--project` implies reading sources from disk; `--source -` for stdin is a `--source`-only feature. [M5 D59, D61.]
- **ImplicitUsings can now be disabled.** `<ImplicitUsings>disable</ImplicitUsings>` in a csproj, or `implicitUsings: false` in `ProjectOptions`, omits Carbide's hidden `Carbide.GlobalUsings.g.cs` document. Bare `Console.WriteLine` then needs an explicit `using System;`. [M5 `DocumentOptions.ImplicitUsings`.]
- **XML parser is hand-rolled.** `@carbide/msbuild-lite` has zero runtime dependencies; the ~200-line walker in `src/xml.ts` handles comments, PIs, CDATA, BOMs, and XML entities. Obscure XML features (DTDs, xmlns processing, etc.) are not supported. [M5 D57.]
- **`BuildResult` / `RunResult` default `schemaVersion = 2`.** Emitted payloads carry the current wire version. Consumers validating against a pinned constant bump to 2 when upgrading past M4. [M5 D62; `Services/{BuildResult,RunResult}.cs`.]

## Documented differences (M6)

- **Allow-list is strict by default.** `@carbide/nuget` refuses any `<PackageReference>` that isn't in the 10-package seed allow-list (`Newtonsoft.Json`, `YamlDotNet`, `CsvHelper`, `Humanizer.Core`, `NodaTime`, `Scriban`, `Handlebars.Net`, `Serilog`, `Serilog.Sinks.Console`, `FluentAssertions`). `--allow-list-mode advisory` warns but resolves; `--allow-list-mode off` disables the gate entirely. This is deliberate policy, not a bug — adding an entry requires a PR with a fixture, per M6 §5 D75. [M6 `packages/nuget/src/allowlist.ts`; `MSNUGET020` / `MSNUGET021`.]
- **Safety refusals skip three content classes unconditionally.** Packages that contain `runtimes/<rid>/native/` (native binaries), `build/*.targets` or `build/*.props` (MSBuild tasks Carbide doesn't execute), or `analyzers/` (not yet supported) are rejected before any bytes reach Roslyn. Codes: `MSNUGET015` / `MSNUGET016` / `MSNUGET017`. [M6 `packages/nuget/src/safety.ts`.]
- **Floating versions are rejected.** `1.*`, `1.2.*`, `*` all throw `MSNUGET001`. Carbide requires either a bare pin (`1.2.3`, treated as ≥ 1.2.3), an exact range (`[1.2.3]`), or a bracketed range (`[1.0.0, 2.0.0)`, `(1.0.0, )`, etc.). This keeps `carbide.lock.json` deterministic and byte-identical across runs. [M6 D73; `packages/nuget/src/version-range.ts`.]
- **Empty groups are first-class.** A `<group targetFramework="net6.0" />` with no `<dependency>` children means "this package supports net6.0 with zero transitive deps." Carbide picks that over a distant non-empty group (e.g. `netstandard1.0` with pre-.NET-Core deps). Before this rule landed, Newtonsoft.Json 13.0.3 resolution tried to drag in `Microsoft.CSharp` from the netstandard1.0 group because the resolver couldn't see the empty net6.0 / netstandard2.0 groups. [M6 `packages/nuget/src/nuspec.ts` `NuspecDependencyGroup`.]
- **Nupkg cache lives at `~/.carbide/nuget-cache/<id-lower>/<version>/`.** Each entry has a `.carbide-meta.json` sidecar carrying the SHA-256. Every cache read verifies the digest; mismatches are treated as cache misses so tampered bytes never reach Roslyn. Overridable via `CARBIDE_NUGET_CACHE_DIR` or `ResolveOptions.cacheDir`. [M6 `packages/nuget/src/cache.ts`.]
- **`carbide.lock.json` is read-first, write-second.** When the CLI runs `build` / `run` / `validate` with `--project`, it looks for a lock next to the csproj. If present, the resolver replays it verbatim (no network, SHA-256 pinned). If absent, it resolves fresh and writes the lock at the end. Pass `--lock <path>` to override the location, `--no-lock-write` to skip writing, `--offline` to forbid network access entirely. [M6 `packages/cli/src/project-file.ts`; §5 D68, D70.]
- **Same-depth version ties pick lexicographically later.** If two distinct packages at the same graph depth request different versions of a shared dep (e.g. A depth-1 wants B@1.0.0, C depth-1 wants B@2.0.0), Carbide keeps the higher string-sort version and emits an `MSNUGET010` warning. Nearest-wins proper (shallower depth beats deeper) runs first. [M6 `packages/nuget/src/resolver.ts`.]
- **MSBLITE013 is suppressed when NuGet resolution runs.** The `@carbide/msbuild-lite` parser still emits `MSBLITE013` for every `<PackageReference>` it captures, but `@carbide/cli` filters it out once it hands the references off to `@carbide/nuget.resolve()` — otherwise every project would log the same warning for packages Carbide is in fact resolving. The underlying warning text was also updated to drop "lands in M6" now that M6 has shipped. [M6 `packages/cli/src/project-file.ts`; `packages/msbuild-lite/src/index.ts`.]
- **Lock replay silently drops packages missing from the lock.** If the `.csproj` declares a `<PackageReference>` that isn't in the lock file, `replayLock()` ignores it and proceeds. Matches npm / yarn `--frozen-lockfile` semantics (trust the lock) but is surprising the first time a user updates the csproj without refreshing the lock. Delete `carbide.lock.json` to force a fresh resolve. [M6 `packages/nuget/src/resolver.ts` `replayLock`.]
- **Packages are compile-time only, never runtime-resolved at load.** The resolved DLLs become `session.addReference` handles before the user's PE is emitted. Mono-WASM's `Assembly.Load(byte[])` dependency resolution is already pre-loaded + handled via `AppDomain.AssemblyResolve` from M3. No new runtime-loading mechanics in M6. [M6 plan §5 D69.]

## Documented differences (M9)

- **`<ProjectReference>` triggers a graph walk.** `carbide build` / `run` / `validate` with `--project` now parses every csproj reachable via `<ProjectReference>` edges, compiles sub-projects in topological order (leaves first), and attaches each producer's PE as a metadata reference for downstream consumers. A single `CarbideSession` covers the whole graph so startup cost is paid once. [M9 `packages/cli/src/project-graph.ts`; D86.]
- **Multi-project build order is deterministic topological; parallel compilation is not used.** Even when two sub-projects have no dependency relationship, M9 compiles them sequentially. A parallel-compile follow-up is noted in the M9 plan §8 but intentionally deferred. [M9 D92.]
- **Per-project `carbide.lock.json` is the only lock-file layout; no aggregated lock.** Each sub-project keeps its own lock next to its own csproj so a monorepo sub-team can commit a library's lock independently of the app's. The CLI's `--lock <path>` flag overrides only the root's lock — sub-projects keep resolving into their own sibling files. [M9 D81.]
- **Sibling-project cycles are a hard error.** `A.csproj → B.csproj → A.csproj` exits 1 with `MSPROJ001: ProjectReference cycle detected: …`. The error message lists every csproj in the cycle. Carbide does not attempt to break cycles; users restructure their graph, same as `dotnet build`. [M9 D80; `MSPROJ001`.]
- **AssemblyName collisions across sub-projects are hard errors.** If two projects in the graph both set `<AssemblyName>Foo</AssemblyName>`, the CLI exits 3 with `MSPROJ002` before any compilation runs. Keeps the flat `--out/` layout unambiguous. [M9 D85; `MSPROJ002`.]
- **`--out -` is rejected for multi-project graphs.** The stdout-piping shape assumes exactly one PE; two would corrupt each other. `--out -` on a root-only graph keeps its M4 semantics. [M9 D84; `MSPROJ003`.]
- **Diagnostics gain a `project` field.** Under `--format json`, every diagnostic now carries `project: <csproj path>` when it originates in a sub-project, or `project: null` (or an omitted field) for the root. The top-level `schemaVersion` stays at 2 because the change is additive. [M9 D91.]
- **Sibling-project PE attachment respects transitive closure.** When building project `P`, Carbide only attaches PEs from projects in `P`'s transitive closure of `<ProjectReference>` targets. This matches the `dotnet build` rule that private dependencies stay private and keeps unrelated siblings out of `P`'s compile-time reference set. [M9 `compileGraphInOrder` / `ProjectNode.transitiveClosure`.]
- **Sub-project PE bytes stay in memory for the whole build.** No filesystem temp cache; each producer's PE is a `Uint8Array` on the shared `CarbideSession`'s reference registry until the session shuts down (or `--out/` writes it out). Avoids introducing a temp-file lifecycle M9 doesn't need. [M9 D83.]
- **MSBLITE014 is suppressed when the graph walker runs.** The `@carbide/msbuild-lite` parser still emits `MSBLITE014` for every captured `<ProjectReference>`, but `@carbide/cli` filters it once the walker consumes the list — otherwise every project would log a warning about references Carbide is in fact building. Mirrors the M6 MSBLITE013 suppression pattern. [M9 `packages/cli/src/project-graph.ts`; `packages/msbuild-lite/src/index.ts`.]
- **`--ref` flags attach only to the root.** CLI `--ref <path>` flags feed extra reference DLLs to the *root* project's compilation; sub-libraries do not inherit them. Keeps the single-project flag semantics unsurprising. [M9 `packages/cli/src/project-file.ts` `runProjectGraphPipeline`.]
- **`validate` now applies `InferOutputKind` to every sub-project.** `Project.getDiagnostics()` used to default to `ConsoleApplication` and raised CS5001 on library projects. With M9's multi-project validate, libraries are routine; Roslyn's output kind is now inferred the same way `build()` infers it, so library projects validate cleanly when they have no `Main`. [M9 `Services/ProjectCompiler.cs` `GetDiagnosticsAsync`.]
- **`.sln` parsing is still not supported.** The CLI's `--project` flag takes a single `.csproj`. Supporting `.sln` would overshoot into Shape S6 territory, which the vision doc excludes. Multi-`.csproj`-with-no-sln remains the ceiling. [M9 D90; vision §7.]

## Documented differences (U1)

- **CLI JSON trailer is framed by a sentinel line.** The default `--format json` mode now emits `\x1E\x1Ecarbide-json\x1E\x1E\n` immediately before the JSON payload so consumers can locate the trailer unambiguously even when user code writes raw bytes via `Console.OpenStandardOutput()`. The legacy "parse the last non-empty stdout line" heuristic remains supported; the sentinel is additive. [U1.1; `packages/cli/src/json-output.ts`.]
- **CLI JSON payloads carry `schemaVersion: 3`.** Every success/failure payload now stamps a top-level `schemaVersion` field. This is distinct from the core-interop `SCHEMA_VERSION` (which remains 2). Consumers that version-pinned against "CLI shape as of M6/M9" read schemaVersion 3 going forward; absent field implies pre-U1. [U1; `packages/cli/src/json-output.ts`.]
- **Default CLI log level is `warning`.** Successful `build`/`run`/`validate` invocations now produce zero stderr output by default. Pre-U1 behaviour (`info` + `trce` + `dbug` lines to stderr) is reachable via `--verbose` / `-v`, `--log-level information`, or `CARBIDE_LOG_LEVEL=info`. [U1.2; `packages/core/src/Hosting/WebAssemblyConsoleLogger.cs` static `WebAssemblyConsoleLoggerConfig.MinLogLevel`.]
- **Every CLI failure carries a structured `error` field.** The payload shape is `{ schemaVersion: 3, success: false, error: { code, category, message, details? }, diagnostics, warnings }`. Stable categories: `internal`, `schema-mismatch`, `flag-error`, `lock-read`, `allow-list-refusal`, `safety-refusal`, `offline-cache-miss`, `nuget-fetch-failed`, `project-graph-cycle`, `assembly-name-collision`, `project-reference-missing`. [U1.3; `packages/cli/src/errors.ts`.]
- **Exit codes 4 and 5 are new.** Exit 4 = NuGet policy refusal (allow-list or safety). Exit 5 = NuGet network / offline cache miss. Previously both bucketed under exit 2. Scripts that branched on exit 2 should either widen the check (treat any non-zero as failure) or split the cases. [U1.3 D101.]
- **`--verbose`/`--quiet` are mutually exclusive.** Combining them exits 3 with `category: "flag-error"`. [U1.2 D98.]

## Documented differences (U2)

- **Program argv is forwarded from `--` to `Main(string[] args)`.** Pre-U2, `carbide run ... -- alpha beta` parsed the args but passed `Array.Empty<string>()` to the user program. U2 forwards them: both `static void Main(string[] args)` and top-level-statements programs (via Roslyn's synthesised wrapper) receive the forwarded array. Programs with a parameterless `Main()` see no args. `Environment.GetCommandLineArgs()` still returns Carbide's own process argv, not the user's forwarded args — use the `Main` parameter. [U2.1; `packages/core/src/Services/ProjectCompiler.cs` RunAsync.]
- **Stdin is eagerly-buffered.** `carbide run ... --stdin <path>` reads the file UTF-8, and `--stdin -` reads the CLI's own stdin to EOF before booting the runtime. The buffered string pre-seeds `Console.In` so the user program can call `Console.In.ReadToEnd()` / `ReadLine()`. When `--stdin` is not passed, `Console.In` stays on the Mono-WASM default (throws `PlatformNotSupportedException` if the user program tries to read it — same as pre-U2, no behavioural change). [U2.2.]
- **`Console.In` is patched via reflection on Mono-WASM.** `Console.SetIn(TextReader)` throws `PlatformNotSupportedException` on browser-targeted WASM. Carbide sets the internal static `_in` (or `s_in`) field directly via reflection when `stdin` is provided. Once set, `Console.In`'s getter returns the cached value without hitting the platform-not-supported path. [U2.2; `ProjectCompiler.SetConsoleInField`.]
- **Interop schema version bumps to 3.** `SCHEMA_VERSION` in `packages/core/src/ts/interop/schema.ts` is 3 as of U2 (was 2 since M5). The C# validator accepts 1 / 2 / 3 during the transition window so pre-U2 TS clients keep working; new TS clients always emit 3. The JSExport `RunAsync` signature gained a second `runOptionsJson` parameter (empty string = defaults). [U2.3 D105.]
- **`carbide run --format json` gains an `invocation` field.** Every run payload (success or failure) now carries `invocation: { args: string[], stdinBytes: number }`. Additive — doesn't bump the CLI wire schema (still 3). [U2 D112.]
- **Stdin is UTF-8 text only.** Binary stdin is not supported. A `--stdin-encoding <enc>` follow-up is out of scope for U2. [U2.2 D109.]
- **`Console.OpenStandardInput()` still bypasses capture.** Just as U1 left `Console.OpenStandardOutput()` writes escaping the SetOut capture, U2 leaves `Console.OpenStandardInput()` reads untouched. Mirror-image limitation; documented for future reference. [U2 §7.]

## Documented differences (U3)

- **`carbide audit --project` is a new read-only subcommand.** Prints the parsed csproj properties, the project graph, per-sub-project source sets, per-sub-project NuGet resolution, and aggregated warnings — without compiling or writing `carbide.lock.json`. Pass `--write-lock` to opt in to writing locks. Useful for debugging `--project` behaviour and pre-commit lint workflows. [U3.1; `packages/cli/src/commands/audit.ts`.]
- **`carbide tree --project` renders the project graph as an ASCII tree.** Root at the top, ProjectReference children indented with `└──` / `├──` connectors, direct PackageReferences as `[nuget] <id> <version>` leaves. Diamond deps are shown once with `<already shown above>` on subsequent encounters. Read-only, like `audit`. [U3.2; `packages/cli/src/commands/tree.ts`.]
- **CS8802 in `--project` mode gets a hint diagnostic.** When Roslyn's "Only one compilation unit can have top-level statements" fires during a `build` or `validate` on a `--project`-driven build, the CLI appends an info-severity `CARBIDE_HINT_CS8802` diagnostic pointing at the three escape hatches (explicit `<Compile Include>`, `--scratch`, or moving statements into a class/method). `--source` mode is untouched. [U3.3 D122–D123; `packages/cli/src/project-file.ts` `attachCs8802Hint`.]
- **`--scratch` permits `--project` + `--source` together.** Without the flag, the M5 D59 mutual-exclusion rule is unchanged (exit 3). With `--scratch`, `--source` files are added to the root project's source set after the csproj-derived files; sub-projects are not affected. Works on `build`, `run`, and `validate`. [U3.4 D124–D125.]
- **`runProjectGraphPipeline` gains `skipLockWrite` and `extraRootSources` options.** Drives the `audit` / `tree` read-only path and the `--scratch` append path respectively. TS-API-direct consumers can opt in to either. [U3 `packages/cli/src/project-file.ts`.]

## Documented differences (M11)

- **`@carbide/msbuild-lite` now evaluates `<Import>` and `Directory.Build.props`.** `parseCsproj` discovers the closest `Directory.Build.props` walking up from the csproj directory and evaluates it before the csproj itself. Explicit `<Import Project="…"/>` elements (with `Condition`) chain through the evaluator in document order. `$(MSBuildThisFileDirectory)` / `$(MSBuildProjectDirectory)` and the rest of the `$(MSBuildThis*)` / `$(MSBuildProject*)` family are substituted in the Project attribute. Nested imports, diamond dependencies (de-duplicated by canonical path), and import cycles (`MSBLITE025`) all resolve. [M11.1–M11.2; `packages/msbuild-lite/src/evaluator.ts`.]
- **`Directory.Build.targets` is auto-discovered but NOT ingested.** Targets files typically hold `<Target>` definitions Carbide can't run; we log `MSBLITE027` (unless the file is an empty `<Project/>` marker — those are silent) and skip the file. Properties authored in a `.targets` file are therefore invisible to Carbide. Users who want those properties to apply should move them into `Directory.Build.props`. [M11 D131; `packages/msbuild-lite/src/index.ts` `targetsFileHasContent`.]
- **Byte-identical PE across the evaluator refactor.** A csproj that inherits `<Nullable>enable</Nullable>` from a parent `Directory.Build.props` produces byte-identical PE bytes to a flattened csproj that declares `<Nullable>enable</Nullable>` directly. Extends the M5 D53 determinism contract across M11's multi-file evaluation. Covered by `packages/cli/test/m11-directory-props.test.mjs`. [M11 §2.6.]
- **Import de-duplication by canonical path.** Upstream MSBuild evaluates each `<Import>` site independently (a file imported twice runs twice). Carbide de-dupes: each canonical path is walked at most once per `parseCsproj`. Matches typical usage (imports are idempotent) and simplifies cycle detection. True cycles (`A → B → A`) still throw `MSBLITE025`. [M11 D129.]
- **Reserved MSBuild properties can't be overridden.** Attempting to set `<MSBuildProjectDirectory>…</MSBuildProjectDirectory>` via `<PropertyGroup>` emits `MSBLITE029` and the write is discarded. Matches upstream MSBuild. [M11 D135; `packages/msbuild-lite/src/reserved-properties.ts`.]
- **`<Choose>/<When>/<Otherwise>` is refused, not emulated.** Emit `MSBLITE023` and continue; use `<PropertyGroup Condition="…">` for the same effect. [M11 D133.]
- **`<Target>`, `<Task>`, `<UsingTask>`, `<ItemDefinitionGroup>` are refused with codes MSBLITE020–022, 028.** Carbide doesn't execute targets / tasks; refusing with a code gives users signal rather than silent ignorance. [M11 §2.5.]
- **`ProjectModel.warnings[].sourceFile` is a new optional field.** Each warning emitted during the evaluator walk records which file (csproj vs. imported .props vs. auto-discovered Directory.Build.props) surfaced it. Additive; existing consumers that only read `{code, message, severity, category}` are unaffected. [M11 D140.]
- **`ProjectModel.evaluationTrace.imports[]` is a new optional field.** Chronological list of every file the evaluator touched — imported, refused, duplicated, or missing. Absent in the pre-M11 shape; additive. [M11 types.ts `ImportTraceEntry`.]
- **Package name stays `@carbide/msbuild-lite`.** The architecture doc's M11 entry envisioned renaming to `@carbide/msbuild`; we keep the `-lite` suffix because the bounded-subset nature is a feature, not a transitional name. [M11 D128.]
- **Parity fixture directories include empty marker files.** `packages/msbuild-lite/test/parity/Directory.Build.props` and `.targets` are intentionally empty `<Project/>` sentinels that stop the ancestor walk at the parity-fixture root, preventing the repo's own `Directory.Build.props` / `.targets` from leaking into snapshot expectations. Real-world Carbide users get the auto-discovery; the parity fixtures deliberately isolate themselves. [M11 `test/parity/Directory.Build.*`.]

## Documented differences (T1)

- **`Project.runInteractive(options)` is a new browser-only run path.** Parallel to `Project.run()`, not a mode flag on it. Takes an xterm.js-shaped `Terminal` instance (any object with a `write(data: string | Uint8Array)` method works; Carbide never imports `@xterm/xterm` directly), returns a `TerminalSession` handle with an `exitPromise` and an idempotent `dispose()`. Calling from a Node-backed session throws synchronously with a helpful message — T1 is the browser's interactive surface, not the CLI's. [T1.1 DT-T1.1; `packages/core/src/ts/project.ts` `runInteractive`.]
- **Interop schema version bumps to 4.** `SCHEMA_VERSION` in `packages/core/src/ts/interop/schema.ts` is 4 as of T1 (was 3 since U2). The C# validator accepts 1 / 2 / 3 / 4 during the transition window. TS-side parsers (`parseRunResult`, `parseBuildResult`) now accept inbound payloads with schemaVersion 3 or 4, so a partially-rebuilt tree (TS built at 4, C# still at 3) fails loudly on real mismatches rather than on the one-step transition. New TS clients always emit 4. [T1.1; `packages/core/src/ts/interop/schema.ts`.]
- **New `RunInteractiveAsync` / `DisposeTerminal` JSExports land alongside existing RunAsync.** The `CompilationInterop` surface now includes both. `DisposeTerminal` is a T1 stub — drain + restore already happen inside `RunInteractiveAsync`'s own `finally` block; T2 will extend `DisposeTerminal` to unblock pending async reads and unwire the input-side of the bridge. [T1.3; `packages/core/src/CompilationInterop.cs`.]
- **Stdout / stderr stream to the bridge in bounded windows, not at end-of-run.** `StreamingStdOutWriter` (used for both streams) coalesces writes into a pooled `char[]` buffer and flushes when the buffer reaches 4 KB or 32 ms since the last flush, whichever trips first. A final drain runs when the entry point returns / throws, so nothing is lost. The TextWriter is not synchronized — Mono-WASM is single-threaded and the interactive run path doesn't share the writer across threads (DT-T1.1). [T1.2; `packages/core/src/Terminal/StreamingStdOutWriter.cs`.]
- **`RunResult.stdOut` and `stdErr` are populated even in interactive mode.** The C# side tees each flushed chunk into a `StringBuilder` as well as the JS bridge, so `await session.exitPromise` resolves with a `RunResult` carrying the full transcript — same shape as `project.run()`. Tests can assert on `runResult.stdOut` without reading back from xterm. Stderr's tee captures raw text; the SGR wrap (`stderrStyle: "plain" | "dim" | "red"`, default `"plain"`) is applied only on the JS-bound path. [T1.4; `packages/core/src/Services/ProjectCompiler.cs` `RunInteractiveAsync`.]
- **`HostAdapter.resolveRuntimeConfigOverlays?()` is a new optional interface method.** Non-breaking for existing adapter implementations because the method is optional. `BrowserHostAdapter` implements it to return `{ print, printErr }` multiplexers that route through the attached terminal sink when one is active and fall back to `console.log` / `console.error` otherwise. `bootRuntime` merges the returned overlay into `DotnetHostBuilder.withConfig(...)` once per boot. [T1.6 DT-T1.5; `packages/core/src/ts/host/browser/browser-adapter.ts`.]
- **`Console.OpenStandardOutput()` bypass is closed for interactive mode.** Pre-T1, bytes written through the handle-level stdout stream went to `console.log` via emscripten's default `print`, bypassing `Console.SetOut` entirely (the U1 drift footnote on the same issue). The T1 `print` overlay routes them into `terminal.write` when a session is attached; the U1 footnote no longer applies to `runInteractive` callers. `project.run()` is unchanged — it still sees the pre-T1 behaviour because no sink is attached during non-interactive runs. [T1.3 DT-T1.6; `packages/core/src/ts/host/browser/browser-adapter.ts` `resolveRuntimeConfigOverlays`.]
- **`globalThis.Carbide.Terminal` is the bridge namespace.** `installBridge(terminal)` sets `globalThis.Carbide.Terminal = { write, writeErr }`; `uninstallBridge()` deletes the key on teardown. The C# side's `[JSImport("globalThis.Carbide.Terminal.write")]` / `.writeErr` imports resolve against those globals and are never invoked outside a live session. At most one interactive session per project at a time — a second concurrent `runInteractive` on the same project throws. [T1.5 DT-T1.8; `packages/core/src/ts/terminal/bridge.ts`.]
- **Synchronous `Console.ReadLine` / `Console.In` still disconnected in T1.** T1 is output-only. `Console.In` inside an interactive run behaves identically to `project.run()` with no `stdin` option: the default Mono-WASM reader throws `PlatformNotSupportedException`. Line-mode and key-mode input land in T2. [T1 §7 `out of scope`.]
- **Writes after the entry point's `Task` completes may race the drain flush.** If user code spawns a fire-and-forget `Task` that writes to `Console.Out` after `Main` returns, those writes can arrive between the drain flush and the `Console.SetOut(old)` restore. Inside that window they still go through the streaming writer; after the restore they go to the pre-T1 default (the previous `Console.Out`). Matches `project.run()`'s pre-existing semantics for late writes. [T1 §6 RT-T1.3.]
- **xterm.js is a structural peer, not a direct dependency.** `@carbide/core`'s `package.json` lists no hard dependency on `@xterm/xterm`. The `InteractiveRunOptions.terminal` type is a structural `XtermTerminalLike` (a `write(data)` method) so the host page chooses its xterm version and Carbide's imports stay framework-agnostic. [T1 DT-T1.10; `packages/core/src/ts/types.ts` `XtermTerminalLike`.]

## Documented differences (T2)

- **`CarbideConsole` public API lives at `Carbide.Terminal.CarbideConsole`.** Covers the primary interactive surface for code Carbide compiles from source: `ReadLineAsync` / `ReadKeyAsync` (async input), `ForegroundColor` / `BackgroundColor` / `ResetColor` (SGR emission), `SetCursorPosition` / `CursorVisible` (cursor), `Title` (OSC 0) / `Clear` (ED+CUP home), `WindowWidth` / `WindowHeight` / `BufferWidth` / `BufferHeight` (geometry cache), `TerminalResized` + `CancelKeyPress` events, `TreatControlCAsInput` + `RunCancellationToken`, `DelayAsync` (JS-setTimeout-backed delay) and `WriteRaw` (escape hatch). Available only inside an interactive terminal session; members throw `InvalidOperationException` outside. [T2; `packages/core/src/Terminal/CarbideConsole.cs`.]
- **`Console.In` is installed as `BrowserTerminalReader` for interactive runs.** The reflection-patched `Console._in` path (from U2) now points at a `BrowserTerminalReader` for the duration of every `RunInteractiveAsync`. `Console.In.ReadLineAsync()` resolves from the same queue `CarbideConsole.ReadLineAsync()` uses. Synchronous `Console.In.Read` / `ReadLine` / `ReadToEnd` / `Peek` / `ReadBlock` throw `NotSupportedException` with a pointed message ("would block the Mono-WASM main thread and deadlock the xterm event pump — use the Async variant"). Matches the parent-plan DT-T2.2 contract. [T2; `packages/core/src/Terminal/BrowserTerminalReader.cs`.]
- **`System.Console.ReadKey` remains PNS on browser.** T2's `CarbideConsole.ReadKeyAsync` is a new API; the native `Console.ReadKey` in the Mono-WASM BCL still throws `PlatformNotSupportedException`. T3's forked `System.Console.dll` replaces it. [T2 out-of-scope; see T3 in the parent plan.]
- **JS line editor handles local echo in line mode.** The editor echoes printable chars back to xterm, handles backspace (`\x7F` / `\b` → `\b \b`), commits on `\r` or `\n`, swallows most other control bytes. Arrow keys are echoed through as-is. Unknown escape sequences get swallowed pending further editing-feature work. [T2; `packages/core/src/ts/terminal/line-editor.ts`.]
- **`KeyParser.cs` ported verbatim from `lib/dotnet/runtime`.** Used by `CarbideConsole.ReadKeyAsync` to decode xterm's key escapes into `ConsoleKeyInfo`. The `TerminalFormatStrings` dependency is replaced with a minimal `XtermTerminfo` shim (xterm-256color only, `IsRxvtTerm=false`); the stray POSIX-termios dependencies are hard-coded (`posixDisableValue=0xFF`, `veraseCharacter=0x7F`). Signature stays byte-compatible with upstream so a future resync is a clean merge. [T2 DT-T2.4; `packages/core/src/Terminal/KeyParser.cs`.]
- **Interop schema version bumps to 5.** `SCHEMA_VERSION` in `packages/core/src/ts/interop/schema.ts` is 5 as of T2 (was 4 since T1). The C# validator accepts 1/2/3/4/5 during the transition window. New JSExports: `DeliverStdIn`, `NotifyResize`, `DeliverSignal`, `SetTreatControlCAsInput`. New JSImports: `setKeyMode`, `setTreatControlCAsInput`, `delay`. [T2; `packages/core/src/CompilationInterop.cs`.]
- **`CarbideConsole.DelayAsync` routes via `globalThis.Carbide.Terminal.delay` (setTimeout-backed).** `Task.Delay(int)` on Mono-WASM browser throws `PlatformNotSupportedException: Cannot wait on monitors on this runtime` because continuations dispatch to an absent thread pool. `CarbideConsole.DelayAsync` uses a JS Promise from `setTimeout` instead, which is the canonical Mono-WASM async-delay pattern. [T2 DT-T2.*; `packages/core/src/Terminal/CarbideConsole.cs` `DelayAsync`.]
- **T2.1 — known runtime regression blocks three fixtures.** On Mono-WASM browser-wasm, awaiting a `Task` that was completed from a JSExport-triggered path (event handler, CT callback, or inner async-method await) currently trips `PlatformNotSupportedException: Cannot wait on monitors`, even when a single-threaded `SynchronizationContext` is installed. The three fixtures that exercise those paths (`interactive-readkey`, `interactive-ctrlc`, `interactive-resize`) are `test.skip`ped with inline TODO comments. The three passing T2 fixtures (`interactive-readline`, `interactive-color`, `interactive-sync-throw`) cover the surfaces that *do* work today (TCS-backed TextReader, SGR emission, pointed-throw on sync reads). T2.1 is a runtime-workstream follow-up: either install a proper Blazor-style `WebAssemblyDispatcher` / `SynchronizationContext` that survives JSExport re-entrance, or wait for T3's forked `System.Console.dll` to bring its own well-tested async scheduling. [T2 follow-up; tracked as T2.1.]
- **Synchronous `CarbideTerminalInterop.WriteStdOut` / `WriteStdErr` JSImports are idempotent at disposal.** T2 doesn't add a new teardown step beyond T1's; `TerminalInputState.Dispose` is called via `InputStateDisposer` in `ProjectCompiler.RunInteractiveAsync`'s `using` chain, pulling the state out of the projectId→state registry and completing the reader (so a pending async read wakes with EOF). [T2; `packages/core/src/Terminal/TerminalInputState.cs`.]
- **`StreamingStdOutWriter` overrides `FlushAsync` / `WriteAsync` / `WriteLineAsync` to run synchronously.** The default `TextWriter.FlushAsync` dispatches to `TaskScheduler.Default` (thread pool), which throws on single-threaded Mono-WASM browser. The override performs the same work synchronously and returns `Task.CompletedTask`. User code that calls `await Console.Out.FlushAsync()` works unmodified. [T2 fix; `packages/core/src/Terminal/StreamingStdOutWriter.cs`.]
- **Carbide installs a minimal `SynchronizationContext` at `InitAsync`.** `CarbideSyncContext` runs `Post` callbacks inline on the single-threaded browser-wasm main thread. Fixes a whole class of "continuation scheduler is missing" errors that otherwise surface as `Cannot wait on monitors`; does not (alone) fix the T2.1 JSExport re-entrance case. [T2 fix; `packages/core/src/Terminal/CarbideSyncContext.cs`.]
