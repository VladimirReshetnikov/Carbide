## Summary

Carbide already has a coherent architecture and a surprisingly broad feature surface for a young codebase: the core runtime, CLI, bounded MSBuild evaluator, NuGet resolver, and interactive browser terminal all fit together cleanly at a high level. The main risks I found are not "missing implementation" problems so much as narrow pieces of logic that undermine the documented contracts around security boundaries, target-framework selection, version resolution, interactive-session lifetime, and build/run consistency.

The most urgent issue is a path-traversal bug in the Node asset server. The next tier of problems are silent semantic drift bugs: `TargetFrameworks` selection no longer follows declaration order, same-depth NuGet conflicts are resolved lexicographically rather than semantically, and `project.build()` does not treat classic `Main` entry points as console apps. The interactive terminal code also exposes a lifecycle contract that the current implementation does not honor.

I started from the docs first, then moved into the source:

1. Read repository-level guidance (`AGENTS.md`, root `README.md`, `docs/agent-workflows.md`, `TECHNICAL_DOCUMENTATION_STANDARD.md`).

## Findings

### 1. High — Node asset server traversal guard can expose sibling directories that share the root prefix

- Evidence:
  - `src/Carbide/packages/core/src/ts/host/node/asset-server.ts:54-58`
- What happens:
  - The server resolves `abs = path.resolve(rootDir, rel)` and then treats `abs.startsWith(rootAbs)` as the traversal check.
  - Prefix checks are not path-boundary checks. A root like `C:\work\_framework` also prefixes `C:\work\_framework-backup\secret.dll`, so requests that escape upward and land in a sibling directory with the same prefix survive the guard.
- Why this matters:
  - Carbide intentionally spins up localhost asset servers for runtime assets and ref-pack assets.
  - A traversal bug here widens the server from "serve this package directory" to "serve arbitrary sibling trees whose path starts with the same bytes", which is a real local-file disclosure risk.
- Recommendation:
  - Replace the prefix check with a separator-aware containment check, for example by computing `path.relative(rootAbs, abs)` and rejecting paths that are absolute or start with `..`.
  - Add an explicit regression test that exercises a sibling directory such as `root` versus `root-evil`.

### 2. Medium — `TargetFrameworks` selection no longer follows the documented first-listed rule

- Evidence:
  - `src/Carbide/packages/msbuild-lite/src/index.ts:146-154`
  - `src/Carbide/packages/msbuild-lite/README.md`
  - `src/Carbide/docs/Carbide-Current-State-Guide.md`
- What happens:
  - `ctx.tfms` is deduplicated and then sorted: `const uniqueTfms = [...new Set(ctx.tfms)].sort();`.
  - The selected framework is then `uniqueTfms[0]`.
  - That means declaration order is discarded before the "selected TFM" decision is made.
- User-visible consequence:
  - A project declaring `<TargetFrameworks>net8.0;net10.0</TargetFrameworks>` is reported and consumed as if `net10.0` were first.
  - That selection feeds downstream behavior such as CLI `ProjectOptions`, NuGet resolution, and cross-project compatibility warnings.
- Why this matters:
  - The docs repeatedly say "first-listed wins".
  - This is exactly the kind of bounded-MSBuild semantic that users will depend on when they trim projects down to Carbide's supported subset.
- Recommendation:
  - Preserve original order while deduplicating, instead of sorting.
  - Add a regression test that asserts both the candidate order and the selected TFM for at least one reversed list (`net8.0;net10.0`).

### 3. Medium — same-depth NuGet conflicts are resolved with `localeCompare`, not semantic version ordering

- Evidence:
  - `src/Carbide/packages/nuget/src/resolver.ts:91-100`
- What happens:
  - When two same-depth dependencies resolve the same package id to different versions, the resolver decides which version is "newer" with `version.raw.localeCompare(existing.package.version) > 0`.
  - That is string ordering, not NuGet version ordering.
- Failure mode:
  - Multi-digit versions compare incorrectly (`10.0.0` sorts before `2.0.0` lexicographically).
  - Pre-release labels are also compared as raw strings instead of through the package's own parsed version model.
- Why this matters:
  - This code sits on the only path that resolves equal-depth conflicts.
  - A wrong winner here means Carbide can build against the wrong package graph even when all inputs are otherwise valid.
- Recommendation:
  - Reuse the already-imported semantic comparer (`compareVersion`) on parsed versions rather than comparing raw strings.
  - Extend the resolver tests with a same-depth tie that uses at least one multi-digit major version.

### 4. Medium — `TerminalSession.dispose()` promises a shutdown barrier that the implementation does not provide

- Evidence:
  - Contract: `src/Carbide/packages/core/src/ts/types.ts:151-160`
  - Implementation: `src/Carbide/packages/core/src/ts/terminal/session.ts:63-89`
  - C# side: `src/Carbide/packages/core/src/Services/SessionSolutions.cs:169-179`
- What happens:
  - The public contract says `dispose()` is safe mid-run and "awaits the in-flight run to finish".
  - The actual implementation just tears down the JS bridge immediately and returns.
  - The C# `DisposeInteractive(projectId)` hook that should deliver the teardown signal is still a no-op.
- Consequences:
  - `await terminalSession.dispose()` is not a reliable barrier for "the interactive program has stopped".
  - Mid-run disposal races the still-running program against a removed bridge, which is exactly the case the public API claims to handle.
  - This is especially risky now that T2/T3 add more long-lived interactive state (`Console.In`, resize events, `CancelKeyPress`, forked `System.Console` shims).
- Recommendation:
  - Make `dispose()` await the active run's completion path.
  - Implement a real C# teardown signal in `DisposeInteractive` that completes pending async reads and marks the terminal state as disposed before the JS bridge is removed.
  - Add a browser test that disposes a long-running interactive session before the program exits.

### 5. Medium — `project.build()` emits DLL output for classic `Main`-based console apps

- Evidence:
  - `src/Carbide/packages/core/src/Services/ProjectCompiler.cs:282-295`
  - `src/Carbide/packages/core/src/Services/ProjectCompiler.cs:305-315`
- What happens:
  - `InferOutputKind` only checks for top-level statements.
  - If the source set has an explicit `static Main(...)` but no top-level statements, `BuildAsync()` switches the compilation to `OutputKind.DynamicallyLinkedLibrary`.
  - `RunAsync()` does not share that inference; it always compiles as `ConsoleApplication`.
- Why this matters:
  - The same source set behaves differently depending on whether the caller uses `build()` or `run()`.
  - A traditional console app can run fine in Carbide but still build to a DLL without an entry point.
- Recommendation:
  - Fold `compilation.GetEntryPoint(...)` or equivalent symbol-level detection into `InferOutputKind`.
  - Add a build-oriented test for an explicit `Main` program, not just a run-oriented test.

## Additional Observations

### Test drift already exists in at least one important area

The `msbuild-lite` tests currently reinforce the sorted-TFM behavior rather than the documented "first-listed wins" rule. That means this bug is not just untested; the tests presently bless the wrong semantics. Fixing the code should be paired with correcting the tests and README wording in the same change.

### The review hotspots are mostly lifecycle and boundary code

The findings cluster around:

- path containment and asset serving,
- graph/evaluator selection policy,
- dependency conflict resolution,
- interactive-session teardown,
- build/run parity.

That is a healthy sign in one sense: the main layering is good, but the narrow seams where components meet now need harder invariants and more targeted tests.

## Suggested Follow-Up Order

1. Fix the asset-server traversal check and add a regression test.
2. Fix `TargetFrameworks` order preservation and update the msbuild-lite tests that currently encode the wrong rule.
3. Replace lexicographic NuGet tie-breaking with semantic version comparison and extend the resolver tests.
4. Repair the interactive dispose contract end-to-end (`types.ts`, TS bridge/session orchestration, C# teardown implementation, browser tests).
5. Make build/run output-kind inference consistent for explicit `Main` entry points.
