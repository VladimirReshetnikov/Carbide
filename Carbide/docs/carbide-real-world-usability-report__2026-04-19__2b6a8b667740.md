## Scope and method

This report focuses on scenarios that are closer to day-to-day production utility code than the existing lightweight fixtures.

I expanded the automated test corpus in two ways:

1. **Core corpus scenarios (offline, deterministic):**
   - Ticket triage workflow across multiple source files, records, enums, LINQ grouping, switch-pattern policy, and date-based SLA breach calculations.
   - Ledger reconciliation workflow across multiple source files, CSV parsing, decimal arithmetic with invariant culture, group-by aggregation, and running-balance projection.

2. **CLI integration scenario (live NuGet, online mode + offline replay):**
   - A project that combines `Newtonsoft.Json` and `YamlDotNet` to parse YAML and emit ordered JSON.
   - Lock-file verification and `--offline` replay behavior.

## Usability findings

### What works well

- **Fast “single-project utility” loop:** The `build` / `run` / `validate` model is straightforward and maps well to scripting and automation scenarios.
- **Session/project API ergonomics are clean:** `addSource`, `build`, `run`, and diagnostics access are intuitive and easy to compose for tooling.
- **Deterministic mindset is visible:** Existing tests plus the new corpus scenarios reinforce reproducibility expectations.
- **NuGet lock replay model is practical:** For allowed packages, Carbide can transition from online resolution to offline deterministic execution.

### Current limitations observed in practical workflows

1. **Environment bootstrapping friction is still significant.**
   Running full `@carbide/core` tests requires both Node and a .NET SDK with wasm tooling; in constrained environments this can block validation quickly.

2. **Project-system coverage is intentionally bounded.**
   This is a design choice, but in real project migration it remains the largest practical gap:
   - partial `.csproj` parity only,
   - project-to-project orchestration still incomplete (warning-based workflows still appear in docs/tests),
   - complex build-graph semantics are not yet first-class.

3. **NuGet safety policy is helpful but restrictive.**
   The allow-list model is good for security posture, but for real teams it creates onboarding friction unless package approval workflows are well documented and fast.

4. **Diagnostics and troubleshooting could be richer for larger projects.**
   For growing multi-file workloads, users benefit from:
   - clearer attribution at project/graph boundaries,
   - stronger guidance for unsupported MSBuild features,
   - higher-level remediation hints (not just raw diagnostics).

5. **No strong “migration assistant” UX yet.**
   For real codebases, success depends on quickly answering:
   - “What in my project is unsupported?”
   - “What is the exact workaround?”
   - “What can be auto-transformed?”

## Shortcomings versus real-life expectations

- **Complex build graphs:** Real applications frequently use many sibling projects, shared props/targets, generated code, and analyzers.
- **Enterprise package footprints:** Teams often depend on packages outside a curated allow-list and need auditable override flows.
- **CI ergonomics:** Users expect one-command environment checks and actionable failure messages when prerequisites are missing.

## Improvement proposals

### 1) “Compatibility audit” command (high priority)

Add a CLI command (for example `carbide audit --project`) that emits:
- unsupported/ignored MSBuild constructs,
- dependency allow-list compliance results,
- projected blockers by severity,
- concrete Carbide-compatible rewrites.

This would sharply reduce migration guesswork.

### 2) Expand real-world corpus as a tracked quality gate

Promote a dedicated corpus lane with scenario tags such as:
- `business-rules`,
- `data-pipeline`,
- `multi-project`,
- `nuget-live`.

Track runtime, pass/fail, and determinism fingerprints over time. Treat regressions in this lane as release blockers for “ready” milestones.

### 3) Improve project-graph UX before broad adoption

When `<ProjectReference>` is not fully supported, provide:
- graph visualization output,
- explicit “manual bridge” suggestions,
- optional assisted multi-stage build scripts generated by CLI.

### 4) Formalize package governance workflow

Pair allow-list safety with a lightweight “policy file” mechanism that supports:
- project-local approved package IDs/versions,
- signed provenance metadata,
- auditable decision trail.

### 5) Add environment doctor checks

A `carbide doctor` command should verify prerequisites and print targeted fixes for common failures (missing .NET SDK, missing wasm-tools, missing browser runtime for browser tests).

## Overall assessment

Carbide is already strong for constrained-environment C# compilation/execution scenarios and has a good architecture trajectory. The next step to unlock broader real-world adoption is less about core compile correctness and more about **operational productization**: project-graph ergonomics, migration diagnostics, and governance tooling around package policy.

The added tests in this change move coverage toward realistic utility-app workflows and provide a better baseline for that next phase.
